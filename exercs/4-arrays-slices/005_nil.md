# `nil` em Go: hands-on

`nil` é o valor zero para tipos que são "referências" a algo. Não é um valor universal como `null` em Java/JS — só faz sentido para **6 tipos específicos**: pointer, slice, map, channel, function, interface.

## 1. Pointer nil

```go
var p *int
fmt.Println(p)  // <nil>
fmt.Println(*p) // panic: nil pointer dereference
```

Pointer não inicializado é `nil`. Desreferenciar quebra. Sempre cheque antes:

```go
if p != nil {
    fmt.Println(*p)
}
```

## 2. Slice nil vs slice vazio (pegadinha clássica)

```go
var s1 []int          // nil slice
s2 := []int{}         // slice vazio (não-nil)

fmt.Println(s1 == nil) // true
fmt.Println(s2 == nil) // false
fmt.Println(len(s1))   // 0
fmt.Println(len(s2))   // 0
```

A boa notícia: `append` funciona nos dois casos:

```go
var s []int
s = append(s, 1) // funciona, mesmo s sendo nil
```

Iterar com `range` em slice nil também funciona (zero iterações). Por isso a convenção idiomática é **retornar nil em vez de slice vazio** quando não há resultados.

## 3. Map nil — leitura ok, escrita panica

```go
var m map[string]int
fmt.Println(m["chave"]) // 0 (zero value), não panica
m["chave"] = 1          // panic: assignment to entry in nil map
```

Diferente do slice. Map precisa de `make`:

```go
m := make(map[string]int)
m["chave"] = 1 // ok
```

## 4. Channel nil — bloqueia para sempre

```go
var ch chan int
ch <- 1   // bloqueia eternamente
<-ch      // bloqueia eternamente
```

Parece bug, mas é útil em `select`: setar um canal pra `nil` o "desliga" daquele select.

## 5. Interface nil — a pegadinha mais sutil

Esta confunde até gente experiente:

```go
type MyError struct{}
func (e *MyError) Error() string { return "erro" }

func doSomething() error {
    var e *MyError = nil
    return e // retorna interface não-nil!
}

func main() {
    err := doSomething()
    fmt.Println(err == nil) // false (!!!)
}
```

Por quê? Uma interface é `(tipo, valor)`. Aqui o tipo é `*MyError` e o valor é `nil` — a interface só é `nil` quando **ambos** são nil. 

Regra prática: nunca retorne uma variável tipada como ponteiro concreto numa função que retorna `error`. Retorne `nil` literal:

```go
func doSomething() error {
    return nil // assim sim
}
```

## Tabela de sobrevivência

| Tipo | Operação em nil |
|---|---|
| pointer | desreferenciar = panic |
| slice | ler/range/append = ok; index = panic |
| map | ler = ok (zero value); escrever = panic |
| channel | send/receive = bloqueia |
| interface | chamar método = panic |

## O modelo mental

`nil` em Go não é "ausência de valor" como em outras linguagens — é o **zero value** de tipos que internamente são headers/descriptors. O comportamento de cada um decorre da estrutura interna: slice é `(ptr, len, cap)` onde ptr nil ainda permite operações que checam len; map é um ponteiro pra hash table onde escrever exige a tabela existir; interface é `(type, value)` e a comparação com nil exige ambos zerados.

Entender o `nil` de Go é entender que cada tipo "referência" tem semântica própria — não existe uma regra única.
